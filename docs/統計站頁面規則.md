# 統計站頁面規則（模板）

統計學實驗室現在有三種頁：**首頁**、**文章頁**、**術語頁**（另加一頁術語索引）。這份檔規定每種頁由哪些區塊構成、資料從哪來、新增一頁要動哪些檔。全站通則（色彩、字級、閱讀面無底色、返回鍵、路由回捲）在 `DESIGN.md`，此處只寫統計站自己的東西。

分工是硬的：**版面在 canvas，內容在資料倉**（`../statistics-lab-data`）。頁面上任何讀者讀得到的句子——開場、區塊說明、術語定義、引註——都來自資料倉的 payload，不寫在 JSX 的字串裡。UI 標籤（「全部」「約 7 分鐘」「進術語表」）是介面不是內容，留在頁面的 `COPY` 常數。判準：**這句話會隨網站內容成長而改寫嗎？** 會，就進資料倉。

## 頁型一：首頁（`src/pages/StatisticsLab.jsx` → `/statisticslab`）

由上而下：

1. `PageShell` 頁首（返回 canvas 根 ← 這是全站唯一該這樣做的統計站頁面，因為它就是本站首頁）＋眉標＋站名＋語言切換。
2. **開場**——`statistics.json` 的 `site.intro`。一段，講這個站在做什麼。
3. **文章**節——`site.sections.articles` 的標題與說明，接文章列。每列＝標題、副標、摘要、日期・閱讀時間・標籤。
4. **術語表**節——`site.sections.glossary` 的標題與說明，接**全部術語**按分層攤開（只印名字，密排，可點），底下一條「進術語表」。
5. **收尾**——`site.note`。出處、固定亂數種子、雙語切換這三件事在這裡交代。

規則：

- **文章少於 3 篇時不出主題 tabs**（`MIN_TABS`）。一個「全部」加一個「統計推論」蓋在一篇文章上面是家具，不是導覽。
- **術語表不是頁尾的一條連結。** 在文章還少的階段它是站上最厚的一層內容，首頁要能讓第一次來的人看見它全部有什麼。等文章多到術語表撐不下再談收合。
- 首頁不做 KPI 大卡、不做只放一個數字的色塊（全站規則）。

## 頁型二：文章頁（`src/pages/statistics/<Slug>.jsx` → `/statistics/<slug>`）

外殼＝`SiteHeader`（返回 → **統計站首頁**）＋ `ArticleLayout` 三欄：左＝其他文章（`ArticleNav`）、中＝閱讀欄、右＝本頁目次。

閱讀欄由上而下：標題／副標（`meta.title`／`summary`）→ `ArticleMeta`（發表日、閱讀時間、標籤）→ 正文。

正文是 `.mdx`，**只寫純 markdown**：圖表以無 import、無 props 的標籤出現（`<LadyTastingTea />`），引註寫 `<Cite id="…">`，術語寫 `<Term id="…">`。頁面元件負責三件 `.mdx` 不知道的事：有哪些圖、圖用什麼參數跑、引註與術語從哪個 payload 解析。**寫下一篇文章＝寫散文**，不是寫 React。

硬規則：

- 圖的參數（含亂數種子）在資料倉的 `figures.json`，不在 JSX 裡寫死。
- `<Cite>`／`<Term>` 的 id 在資料倉 build 時就要能解析，解析不到就 build 失敗；前端不做把關、不做 fallback。
- 數學一律 LaTeX（`$…$`／`$$…$$`），原始碼不打 Unicode 數學字元（`npm run validate:math` 會擋）。

## 頁型三：術語頁（`src/pages/statistics/GlossaryTerm.jsx` → `/statistics/glossary/:slug`）

一個術語一頁，**新增術語不必動這個檔**：內文是資料倉同步過來的 `.mdx`，用 slug 取。

外殼＝`SiteHeader`（返回 → 術語表）＋ `ArticleLayout`（左欄＝其他所有術語）。頁面骨架固定四節：

1. 標頭：術語名、一句話定義（`oneLine`，同一份字串也是 hover 卡上的那句）、又稱。
2. 正文四節（`.mdx` 寫）：**它是什麼／它從哪來／一個真實發生過的例子／常見的誤讀**。
3. 相鄰術語（`related`）。
4. 出現在哪些文章（`usedIn`）——由資料倉掃正文的 `<Term>` 標記反向生成，不手寫，不會過期。

## 頁型四：本站說明（`src/pages/statistics/About.jsx` → `/statistics/about`）

正文＝資料倉 `data/about/about.{zh,en}.mdx`，與文章同一套 `<Cite>`／`<Term>` 規則、同一道斷鏈閘。它交代的是這個站對自己的要求：例子怎麼查證、模擬為何可重跑、術語層怎麼運作、兩個倉庫怎麼分工、版式的來歷。

硬規則：**這一頁只能寫資料倉真的會擋的事。** 一段沒有任何閘門在背後執行的紀律宣言，是裝飾。要在這頁多寫一條承諾，先去 `validate-processed.mjs` 加那道檢查。

## 站內工具列（三種內頁共用 `SiteHeader`）

固定在頂端（sticky）、細、無底色、下緣一條髮絲線。左＝返回本站首頁，右＝中英切換、字級、外觀（色票與紙紋，收在下拉裡）。理由：語言與字級是讀者讀到一半才會伸手的東西，過去它們隨頁首捲走，要改得先捲回頂。閱讀頁只付得起一排鉻件，所以色票與紙紋不平鋪在列上。

## 術語索引（`/statistics/glossary`）

按 `glossaryGroups` 分層印，**組序與組內順序都照資料倉，不 sort**（順序是內容，見 `DESIGN.md`「清單與索引」）。每組印標題＋一句說明。搜尋只過濾不重排；命中的若是例子而非名字，把命中的那句例子一起顯示（有人記得的是那條死鮭魚，不是「多重比較」四個字）。

## 新增一頁要動什麼

新增**文章**：資料倉 `data/articles/<slug>/`（meta、figures、正文中英 `.mdx`）→ `npm run build && npm run validate && npm run sync` → canvas 新增 `src/pages/statistics/<Slug>.jsx`（照 `NullHypothesis.jsx` 的形狀，只換 payload 與圖表清單）。首頁、左欄導覽、術語反向連結全部自己長出來。

新增**術語**：資料倉 `data/glossary/<slug>/`（meta＋中英 `.mdx`）→ 到 `data/glossary-groups.json` 把 slug 放進它該屬的組（**不放就 build 失敗**）→ build／validate／sync。canvas 零改動。

改**首頁的話**：改 `data/hub.json`，不改 JSX。改**本站說明**：改 `data/about/about.{zh,en}.mdx`。

文章的「最後更新」日期不是手寫的：build 從 git 取那篇文章檔案的最後 commit 日期。手記的日期是一個遲早會被忘記的承諾，而頁面上的日期一旦是假的，讀者對這頁其他所有東西的信任也一起沒了。

## 這些規則靠什麼強制

不是靠人記得，是靠資料倉的 `npm run validate`（每條都配了負向測試）：首頁文案缺一角、術語沒歸組、組裡沒人、`<Term>`／`<Cite>` 斷鏈、術語缺定義／例子／引註／英文版、圖表模擬缺亂數種子——任一項都讓 build 紅。canvas 側 `npm run build` 另有四道閘（字型覆蓋、設計代幣、色彩系統、數學記號）。
